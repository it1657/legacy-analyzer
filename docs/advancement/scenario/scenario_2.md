# 시나리오 2: 리소스 여유 O, 인터넷망 사용 불가 (폐쇄망/에어갭)

전제: `scenario_0.md`의 `LlmClient` 추상화가 이미 적용돼 있다는 가정 하에 작성한다.

## 목표

GPU·디스크 등 하드웨어 리소스는 넉넉하지만, 인터넷이 아예 연결되지 않은 폐쇄망/에어갭 서버·PC에 배포한다. 이 시나리오는 다른 두 시나리오와 성격이 다르다 — **"pull"이라는 동작 자체가 성립하지 않는다.** Docker Hub도, `ollama pull`도, Anthropic API도 전부 인터넷이 있어야 되는데 여기선 안 된다. 그래서 이 문서의 핵심은 "무엇을 오프라인으로 반입할 것인가"다.

## Provider 선택 UI(P2)는 해당 없음, 상태 표시만 필요

`plan.md`의 P2(관리자 승인형 provider 선택)는 이 시나리오에도 적용하지 않는다 — Anthropic 자체가 인터넷 불가로 원천 차단이라 고를 대상이 없다. 시나리오 1과 동일하게 P2에서 설계한 로컬 LLM 헬스 상태 컴포넌트만 재사용해 "로컬 LLM 준비 완료/연결 실패"를 표시한다.

## DB 선택: Postgres 유지

시나리오 1과 달리 "경량화"가 목표가 아니라 "리소스 여유"가 전제이므로, H2로 낮출 이유가 없다. 기존 `docker-compose.yml`의 `postgres` 서비스를 그대로 유지한다 — "다중 사용자 여부"(아래 절)가 불확실한 상태에서 H2의 동시 쓰기 제약을 감수할 이유도 없다.

## Anthropic API는 선택지가 아니다

`llm.provider=anthropic`은 인터넷이 있어야 동작하므로 이 환경에서는 원천적으로 쓸 수 없다. 다른 두 시나리오에서는 `local`이 "대안"이지만, 여기서는 **`local`이 유일한 선택지**다. 즉 `scenario_0.md`에서 만든 `llm.provider` 토글은 이 환경에서는 사실상 고정값이 되고, 롤백 개념도 "로컬 LLM 품질이 낮으면 Anthropic으로" 가 아니라 "로컬 모델을 더 큰/다른 것으로 바꾼다" 정도로 제한된다.

## 오프라인 반입 전략

인터넷이 되는 환경에서 준비한 뒤 물리적으로(USB, 사내 파일 서버 등) 반입해야 하는 항목들:

1. **컨테이너 이미지**: 인터넷 되는 곳에서 `docker pull`한 뒤 `docker save -o images.tar app:tag postgres:16-alpine ollama/ollama:0.32.1 chromadb/chroma:1.5.9`로 묶어 반입 → 폐쇄망에서 `docker load -i images.tar`. 또는 폐쇄망 내부에 프라이빗 레지스트리(Harbor 등)를 이미 운영 중이면 거기 미리 올려두고 폐쇄망에서는 그 레지스트리를 가리키게 한다(이쪽이 여러 대에 배포해야 하면 더 낫다).
2. **Ollama 모델**: `ollama pull`도 인터넷이 필요하므로, 인터넷 되는 머신에서 받은 뒤 Ollama가 모델을 저장하는 디렉토리(`~/.ollama/models` 또는 컨테이너의 해당 볼륨)를 통째로 복사해 반입 — 이 방법이 폐쇄망에서 `ollama pull`을 시도하는 것보다 안전하다. 정확한 디렉토리 구조는 Ollama 버전에 따라 달라질 수 있으니, 반입 전에 소스 머신과 대상 머신의 Ollama 버전을 반드시 동일하게 맞춘다(버전이 다르면 모델 blob 포맷이 안 맞을 수 있음 — 미검증, 실제 반입 전 소규모 테스트 권장).
3. **Chroma 이미지/의존성**(RAG 채택 시): Chroma 자체는 컨테이너 이미지 하나로 완결되므로 1번과 동일하게 처리.

## 컨테이너 구성: Compose Profiles

`plan.md`의 원칙대로 `ollama`/`chroma`는 별도 컨테이너로 두고 `profiles: ["llm-rag"]`로 묶는다. 이 시나리오는 Anthropic API 자체가 불가능해 로컬 LLM이 유일한 선택지이므로, `.env`에 `COMPOSE_PROFILES=llm-rag`를 기본으로 넣어 `docker compose up -d`만으로 `ollama`까지 항상 뜨게 한다. `chroma`는 RAG 채택 여부(아래 절)에 따라 반입 단계에서부터 포함할지 뺄지 미리 정한다 — 에어갭 환경은 "일단 다 띄워두고 나중에 토글"이 아니라 반입 시점에 확정하는 편이 안전하다(불필요한 이미지·모델을 반입 절차에 태우지 않기 위해).

## 리소스 여유를 살린 모델 선택

이 시나리오는 GPU/디스크가 넉넉하다는 전제이므로, 시나리오 1(경량, 7b)보다 큰 모델을 쓸 수 있다:

- `qwen2.5-coder:32b` 등 더 큰 코드 특화 모델, 또는 인프라 담당자가 이미 검증해둔 다른 모델(예: 사내에 이미 폐쇄망용으로 검증된 Qwen3 계열 GGUF/세이프텐서가 있다면 그걸 Ollama로 서빙).
- 정확한 모델 선택은 실제 서버의 VRAM/디스크 스펙을 확인한 뒤 결정 — 이 문서에서 특정 모델을 못박지 않는다.

## RAG(Chroma) — 채택 기준은 다른 시나리오와 동일(관찰 후 판단)

`plan.md`/`scenario_1.md`/`scenario_3.md`와 같은 기준을 따른다 — RAG는 "컨텍스트 초과가 실제로 확인되면" 채택하는 것이지, 리소스가 넉넉하다고 미리 켜두는 게 아니다. 오히려 이 시나리오는 시나리오 1(7b)보다 큰 모델(32b 등, 컨텍스트도 보통 더 넉넉함)을 쓰므로, 컨텍스트 초과가 발생할 가능성은 세 시나리오 중 가장 낮을 수 있다 — "리소스가 있으니 RAG도 같이"는 잘못된 근거다.

다만 리소스 여유·낮은 네트워크 노출 리스크 덕분에 **채택하기로 결정했을 때의 도입 비용(Chroma 컨테이너 추가, 볼륨 관리)이 가장 낮은 시나리오**이기는 하다. 채택 시 반영 사항:

- 임베딩 클라이언트는 Ollama 대상(`OpenAiCompatibleEmbeddingClient` → `/api/embeddings`)으로 구현.
- 오프라인 반입 목록에 `nomic-embed-text` 모델도 포함.
- Chroma 데이터 볼륨을 로컬 디스크에 영속화(에어갭이라 클라우드 백업이 안 되므로, 정기 백업 정책은 인프라 담당자와 별도 확인).

## 다중 사용자 여부

이 문서는 "리소스 여유 + 인터넷 불가"만 전제로 하고 동시 사용자 수는 규정하지 않는다. 만약 이 폐쇄망 서버를 여러 명이 동시에 쓴다면, `plan.md`의 RAG 섹션에 있는 "단일 로컬 LLM 인스턴스 직렬화" 병목과 `scenario_3.md`의 동시성 완화책(세마포어 캡 등)을 그대로 적용한다 — 폐쇄망이어도 동시 사용자가 많으면 병목은 똑같이 발생한다.

## 실행 순서

0. (선행) 대상 서버의 정확한 하드웨어 스펙(GPU 모델/VRAM, 디스크 여유) 확인 → 모델 크기 결정.
1. 인터넷 되는 머신에서 이미지·모델을 준비하고 반출용으로 패키징(`docker save`, 모델 디렉토리 복사).
2. 폐쇄망으로 물리 반입(사내 정책에 따른 보안 검수 포함 — 이 부분은 인프라/보안팀 프로세스를 따른다).
3. 대상 서버에서 `docker load` + 모델 디렉토리 복원 → `docker compose up -d`.
4. `llm.provider=local` 고정, `llm.local.url=http://ollama:11434` 등 `scenario_0.md` 설정 적용.
5. (선택) RAG 채택 시 Chroma 반입·연동.

## 검증 방법

- 대상 서버에서 실제로 외부 네트워크 호출 없이(방화벽 아웃바운드 차단 상태에서) 전체 스택이 정상 기동·동작하는지 확인 — "혹시 몰래 인터넷을 타는 부분"이 없는지가 핵심 검증 포인트(예: 헬스체크가 외부 NTP나 DNS를 타는 이미지가 섞여 있지 않은지).
- Ollama 버전 불일치로 모델 blob이 깨지는 경우가 없는지, 반입 후 `ollama list`/실제 추론 호출로 확인.
- 대형 프로젝트로 README 생성 시 리소스 여유를 실제로 체감할 수 있는 속도가 나오는지 확인(시나리오 1의 7b 대비).
