# 시나리오 3: 인터넷망 O, 리소스 충분 — Claude API/자체 호스팅 LLM 선택형

전제: `scenario_0.md`의 `LlmClient` 추상화가 이미 적용돼 있다는 가정 하에 작성한다. 이미 여러 명이 실사용 중인 라이브 서비스에, OpenAI 호환 API를 제공하는 자체 호스팅 LLM 서버가 별도로 구성돼 있는 환경을 전제로 한다.

## 목표

인터넷도 되고 리소스도 있으니, Anthropic API와 자체 호스팅 LLM 서버를 상황에 맞게 선택할 수 있게 한다. 다만 **이미 라이브 서비스**이므로, 다른 두 시나리오와 달리 "처음부터 설계"가 아니라 "지금 도는 걸 안전하게 얹는" 마이그레이션 관점으로 접근한다.

**선택의 주체는 관리자다.** `plan.md`의 P2에서 정한 대로, provider는 서버 전역 설정이 아니라 **관리자가 개별 사용자에게 열어주는 권한**(`LOCAL_LLM_USER` Role)이다. 일반 사용자는 관리자가 열어주기 전까지 Anthropic 고정이고, 권한을 받은 사용자만 분석 시 provider를 고를 수 있다. 이 시나리오의 "선택형"이라는 성격 자체가 P2 설계를 전제로 한다 — 아래 롤아웃 계획도 이 권한 모델을 이용한다.

## 자체 호스팅 LLM 연동

- 대상 서버가 OpenAI 호환 엔드포인트(`/v1/chat/completions`)를 제공한다는 전제라면, `scenario_0.md`의 `OpenAiCompatibleLlmClient`가 그대로 대상으로 삼는 규격이라 별도 어댑터가 필요 없다. 이 전제가 맞는지는 실제 서버 소프트웨어 문서를 확인해야 한다.
- 인증: 서버가 API 키 인증을 지원하면(OpenAI 호환 서버 다수가 `Authorization: Bearer <key>` 방식을 채택) `llm.local.api-key`에 발급받은 키를 채운다. 인프라 담당자에게 다음을 확인해야 한다: (1) API 키 필요 여부·발급 절차, (2) 정확한 서빙 모델명/별칭(요청의 `model` 필드에 그대로 들어감), (3) 네이티브 컨텍스트 길이(RAG 필요성 판단에 씀).
- 네트워킹: LLM 서버는 이 프로젝트의 `docker-compose.yml`에 속한 컨테이너가 아니라 **사내망의 별도 서버**이므로, 랩탑 시나리오에서 썼던 `host.docker.internal` 같은 편법이 필요 없다 — 그냥 사내 DNS/IP로 `llm.local.url=http://<내부주소>:<포트>`를 채우면 된다(단, `app` 컨테이너가 그 사내망에 실제로 도달 가능한지는 배포 환경의 네트워크 정책에 따라 다르므로 확인 필요).

## 무중단 단계적 롤아웃 (라이브 서비스이므로 필수)

1. **1단계 — 코드만 배포**: `scenario_0.md`(+P2) 코드를 배포하되, `LOCAL_LLM_USER` Role을 아무에게도 부여하지 않은 상태로 시작. 이 시점엔 모든 사용자가 여전히 Anthropic만 쓰고, 동작 변화가 전혀 없다.
2. **2단계 — 인프라 확인**: 대상 서버에 GPU가 실제로 있는지, 배포가 수동 SSH인지 CI/CD인지, 현재 동시 접속 패턴이 어느 정도인지 확인(현재 미확인 상태 — 계획을 구체화하려면 이 정보가 필요).
3. **3단계 — 제한적 시험**: 전체 사용자에게 한 번에 열지 않고, 관리자 화면(P2)에서 관리자 본인 계정이나 소수 테스트 사용자에게만 `LOCAL_LLM_USER` 권한을 부여해 품질·속도를 검증한다("테스트베드" 성격) — P2의 권한 모델이 그대로 이 단계적 롤아웃 수단이 된다.
4. **4단계 — 확대 또는 유지**: 시험 결과가 괜찮으면 권한을 받는 사용자 범위를 점진적으로 넓히거나, 필요하면 전체 사용자 기본값으로 확대(그 경우에도 "선택 가능"이지 강제 전환은 아님). 문제가 있으면 권한을 회수하고 Anthropic만 남긴다.
5. **롤백**: 개별 사용자 단위로는 Role만 회수하면 즉시 되돌아간다. 로컬/사내 LLM 백엔드 자체에 문제가 생겨 전면 차단해야 하는 경우를 대비해 `LLM_PROVIDER`/게이트 스위치(`scenario_0.md`에서 배선)도 유지 — 이건 "로컬 옵션 자체를 껐다 켰다" 하는 최후 수단이고, 평상시 개별 전환은 Role 부여/회수로 처리한다.

## 다중 사용자 동시성

이미 여러 명이 쓰고 있는 서비스이므로, `LOCAL_LLM_USER` 권한을 받는 사용자가 늘어날수록(3~4단계) 이 문제가 점점 현실이 된다.

- 자체 호스팅 서버 소프트웨어에 따라 동시 요청 처리 능력이 크게 다르다(단일 요청 순차 처리만 지원하는 것부터, 여러 요청을 묶어 처리하는 것까지) — 어느 쪽이든 무한하지는 않으므로, `plan.md` RAG 섹션에서 짚은 "단일 인스턴스 직렬화" 병목은 완전히 사라지는 게 아니라 정도의 차이로 봐야 한다.
- 완화책: (a) 앱 레벨에서 로컬 모드 동시 분석 세션 수를 세마포어로 캡, (b) LLM 서버 쪽 동시 요청 처리 설정을 인프라팀과 협의, (c) 3단계(제한적 시험) 동안 실제 동시 부하를 관찰해서 임계치를 정한다.

## 네트워크 노출

이 시나리오는 리스크가 서로 독립된 두 축으로 갈린다 — "동시 사용자 수"(이미 다중 사용자로 확정)와 "네트워크 노출 범위"(폐쇄망이냐 공용망이냐, 실제 배치에 따라 다름). LLM 서버 자체에 인증(API 키)이 있다는 전제라면 포트가 사내망에만 열려 있는 한 위험은 낮지만, 혹시라도 그 서버가 `app`과 같은 퍼블릭 서브넷에 있고 인터넷에서 직접 닿을 수 있다면 API 키 유무와 무관하게 노출 자체를 막아야 한다(방화벽/보안그룹 확인 필요 — 인프라팀 확인 사항).

## 컨테이너 구성: Compose Profiles

이 시나리오는 다른 둘과 달리 **로컬 `ollama` 컨테이너가 없다** — LLM은 사내/외부의 별도 LLM 서버를 URL로 호출하는 것이지, 이 프로젝트의 `docker-compose.yml`에 서비스로 들어오는 게 아니다. 그래서 `plan.md`의 `profiles` 원칙이 여기선 **RAG 채택 시의 `chroma`에만** 적용된다:

```yaml
services:
  postgres: ...
  app: ...
  chroma:
    profiles: ["llm-rag"]   # RAG 채택 전까지는 정의만 해두고 기본 off
```

이미 라이브 서비스이므로 기본 기동 동작(`docker compose up -d` → postgres+app만)을 바꾸지 않는 게 원칙이다 — `COMPOSE_PROFILES`를 기본으로 켜두지 않고, RAG 채택이 확정된 뒤에만 배포 스크립트/환경변수로 `llm-rag` 프로필을 명시적으로 켠다.

## RAG(Chroma) — 조건부, 서빙 모델의 컨텍스트 확인이 선행

최근 오픈소스 LLM들은 모델·버전에 따라 네이티브 컨텍스트가 수십만 토큰 수준까지 커진 경우가 많다. 정확히 서빙 중인 모델·버전의 컨텍스트 길이를 확인하기 전까지는:

- 컨텍스트가 충분히 크면(대략 이 프로젝트의 `projectStructureSummary`가 넘길 일이 거의 없는 수준이면) **RAG 자체를 도입하지 않는 게 맞다** — 불필요한 인프라(Chroma)와 복잡도를 늘릴 이유가 없다.
- 확인 결과 실제로 대형 프로젝트에서 컨텍스트 초과가 재현되면, 그때 `plan.md`의 RAG 설계를 채택한다.

→ **다음 액션**: 인프라팀에 (1) 정확한 서빙 모델명, (2) 실제 설정된 컨텍스트 길이(예: 실행 옵션의 최대 시퀀스 길이 설정값)를 확인 요청.

## 실행 순서

0. (선행) 인프라 확인 3종 세트 — GPU 유무, 배포 방식, 현재 동시 접속 패턴. LLM 서버 쪽 확인 2종 — API 키 필요 여부, 정확한 모델명/컨텍스트 길이.
1. `scenario_0.md` 코드를 무중단 배포(권한 미부여 상태 = 기존과 동일 동작).
2. P2(관리자 권한 부여 UI, `LOCAL_LLM_USER` Role, `llm.local.*` 환경변수 배선) 적용.
3. 관리자 계정/소수 테스트 사용자에게만 권한 부여 → 품질·속도·동시성 관찰.
4. 컨텍스트 초과 사례 관찰 결과에 따라 RAG 도입 여부 결정.
5. 결과가 괜찮으면 권한 범위를 단계적으로 확대, 문제 있으면 소수 테스트 상태로 유지하거나 회수.

## 검증 방법

- 1단계 배포 후 기존 사용자 트래픽에 아무 영향이 없는지(회귀 없음) 확인 — 가장 중요.
- 시험 전환 중 `LLM_PROVIDER` 롤백이 실제로 무중단으로 되는지 리허설.
- 여러 세션을 동시에 돌려 LLM 서버 응답 지연이 허용 범위인지 실측.
- 서빙 모델의 실제 컨텍스트 길이 확인 후, 이 프로젝트의 대형 프로젝트 케이스가 그 안에 들어오는지 계산으로 먼저 검증(실제 RAG 구현 전에 계산으로 먼저 걸러낼 수 있음).
